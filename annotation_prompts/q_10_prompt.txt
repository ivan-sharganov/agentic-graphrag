SEMANTIC CONSTRAINT
Interpret “documents to bring” strictly as:
— a document was requested/checked during a passport control event.
That is, look for the chain:
Encounter → hasDocumentCheck → DocumentCheck → requestedDocType → DocumentType
(use the exact predicate/label names FROM THE SPECIFIC Cypher!)

DO NOT INCLUDE

- ScreeningStatus, BehaviorGuideline
- Any “advice”, “usually”, “always” — unless it is implemented in the graph as DocumentCheck.
- Any documents not linked via DocumentCheck.

CONTROL STAGE VALIDITY (important)
If an Encounter has controlStage and it does NOT refer to passport control at the desk (e.g., AtGate / AtCheckin and similar), then do NOT include DocumentCheck from that Encounter.
If controlStage is absent, you may assume it is passport control, but only if the Cypher explicitly has DocumentCheck for that Encounter.

PROCEDURE FOR EACH cypher_i.txt
Step 1. Find all Encounter nodes present in the file (usually MERGE (n:Encounter {key:...})).
Step 2. For each Encounter, verify that it is passport control (see the controlStage rule).
Step 3. For valid Encounter nodes, find all DocumentCheck linked to it (Encounter -> hasDocumentCheck -> DocumentCheck).
Step 4. For each DocumentCheck, find its document type (DocumentCheck -> requestedDocType -> DocumentType).
Step 5. Build triples minimally:
  A) evidence: DocumentCheck -> requestedDocType -> DocumentType
  B) connection: Encounter -> hasDocumentCheck -> DocumentCheck
  C) scope: add minimal triples proving Stuttgart scope, only if they are present in this Cypher:
     - Encounter -> atCity -> City
     - Encounter -> atAirport -> Airport
     - Encounter -> atCountry -> Country
     - Airport -> locatedInCity -> City
     - City -> locatedInCountry -> Country
IMPORTANT: add scope triples only if they can connect the Encounter to Stuttgart/Germany (e.g. entityName="Stuttgart" or key="city_stuttgart" appears in Cypher). Semantic matching is allowed (e.g., airport ↔ city from the list), even if there is no explicit link locatedInCity/locatedInCountry.

Step 6. Remove anything extra. Triples must be sufficient but minimal.

HOW TO IDENTIFY “Stuttgart” IN CYPHER
You may treat scope as Stuttgart only if the file contains one of the explicit markers:
	•	City.key == "city_stuttgart" OR City.entityName == "Stuttgart"
	•	Airport.key == "airport_stuttgart" OR Airport.entityName == "Stuttgart Airport" (IATA: STR)
And the Encounter is connected via atCity/atAirport (and if needed locatedInCity) to this node.
