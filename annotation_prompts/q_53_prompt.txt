Ты — генератор аннотаций триплетов (gold facts) для GraphRAG.

ЦЕЛЬ
Сформировать датасет пар (question, triples), где triples — это:
1) Триплеты, которые прямо отвечают на вопрос (evidence triples),
2) Триплеты, которые связывают evidence с корневым узлом Encounter (connection triples),
3) Триплеты, которые доказывают scope (города Италии) — только если такие связи явно присутствуют в Cypher (scope triples).

ВОПРОС
В каком городе италии чаще всего просят показать бронь жилья?

ИСТОЧНИК ИСТИНЫ
Единственный источник истины — текст Cypher-файла. Никаких догадок, знаний о мире, расширений. Если чего-то нет в Cypher — не добавляй.

ОГРАНИЧЕНИЕ НА СЕМАНТИКУ
Ищем только случаи, где запрашивали бронь жилья: documentType = 'hotel_booking' или 'accommodation_proof'.

НЕ ВКЛЮЧАТЬ
- Questioning / questionTopic / wasAsked (кроме вопросов про темы вопросов)
- Stamping / Outcome / resultCode
- ScreeningStatus, BehaviorGuideline
- Любые “советы”, “обычно”, “всегда” — если это не реализовано в графе как DocumentCheck.
- Любые документы, которые не связаны через DocumentCheck.

ВАЛИДНОСТЬ СТАДИИ КОНТРОЛЯ (важно)
Если в Encounter есть свойство controlStage и оно НЕ относится к паспортному контролю у стойки (например AtGate / AtCheckin и аналоги), то DocumentCheck из этого Encounter НЕ включай.
Если controlStage отсутствует — допускай, что это паспортный контроль, но только если в Cypher явно есть DocumentCheck у Encounter.

ПРОЦЕДУРА ДЛЯ КАЖДОГО cypher_i.txt
Шаг 1. Найди все Encounter-узлы, которые присутствуют в файле (обычно MERGE (n:Encounter {key:...})).
Шаг 2. Для каждого Encounter проверь, что он относится к passport control (см. правило про controlStage).
Шаг 3. Оставь только Encounter, которые можно связать с Italy через atCountry или через atCity/atAirport + locatedInCountry; для каждого такого Encounter сохрани связанный City (или Airport→City).
Шаг 4. Для валидных Encounter найди все DocumentCheck, связанные с ним (Encounter -> hasDocumentCheck -> DocumentCheck).
Шаг 5. Для каждого DocumentCheck найди его document type (DocumentCheck -> requestedDocument -> DocumentInstance и DocumentInstance.documentType).
Шаг 6. Построй triples минимально:
  A) evidence: DocumentCheck -> requestedDocument -> DocumentInstance
  B) evidence: DocumentInstance -> documentType -> "..."
  C) connection: Encounter -> hasDocumentCheck -> DocumentCheck
  D) scope: Encounter -> atCity / atCountry / atAirport и связанные locatedInCity / locatedInCountry, если есть.

Допускается смысловое сопоставление (например аэропорт ↔ город из списка), даже если нет явной связи locatedInCity/locatedInCountry.

ФОРМАТ ВЫХОДА (строго)
JSON объект с двумя полями: question и triples.

- question: строка ровно как в задании.
- triples: массив объектов, каждый объект = один триплет.

Каждый триплет:
{
  "s": {"label": "...", "key": "..."},
  "p": "...",
  "o": {"label": "...", "key": "..."}
}

Требование форматирования:
- В каждом триплете поля "s", "p", "o" должны быть на отдельных строках (как в примере).
- Значения внутри "label"/"key"/"p" — в одну строку.

ШАБЛОН (не меняй структуру)
{
  "question": "В каком городе италии чаще всего просят показать бронь жилья?",
  "triples": []
}

ПРИМЕЧАНИЕ
Используй РОВНО те label/predicate/key, которые реально присутствуют в cypher_i.txt.
Документы представлены как:
DocumentCheck -> requestedDocument -> DocumentInstance,
а тип документа — строковое свойство DocumentInstance.documentType.
Поэтому в аннотациях добавляй триплеты:
- DocumentCheck -> requestedDocument -> DocumentInstance
- DocumentInstance -> documentType -> "..."
Строковое значение documentType обязательно включай.
